# SOLA v5.3.35

`memory_leak_validator` — the dedicated validator class deferred from v5.3.24. Closes the static-corpus gap for the carryover-memory threat surface that the jailbreak suite has been catching live since v5.3.24.

## Why this exists

In v5.3.24 I drafted three validator-corpus fixtures for memory-leak threats (false-memory narration, other-learner naming) but had to drop them because no validator class existed that could pattern-match those threats without an LLM round-trip. The v5.3.24 release notes called it out explicitly as a gap. The jailbreak suite covers the same threat against a real provider, so the live-attack guard was already in place — but the static corpus is faster, cheaper, and runs in CI on every push, so it's the right place for repeated regression checks.

This release adds the validator class, re-adds the corpus fixtures, and pins the false-positive cases that any future regex tweak must keep passing.

## What the validator catches

Two threat surfaces, both hard-fail (`SEVERITY_FAIL`):

**False memory narration** — phrases that claim continuity the model does not have. The classifier processes one turn at a time; any reference to a prior SESSION (not just earlier in the current chat) is fabricated. Patterns include:

- `in (our|the) (last|previous|prior|earlier) (session|chat|conversation)`
- `(I |we )?(remember|recall) (from|that|our|when|you said)`
- `based on (our|your) (previous|past|prior|earlier) (conversation|chat|session)`
- `you (mentioned|said|told me|asked) (in|on|during|earlier|before|previously|last time|yesterday|last week)`
- `I (have|'ve) (access to|seen|reviewed) (your|our) (conversation|chat|session) history`

**Cross-learner state leak** — phrases that claim knowledge of other learners. The carryover-memory feature stores notes per-learner; the model must never expose a peer's state. Patterns include:

- `other (students|learners|users|classmates) (in this|are|have|find|tend|struggle|...)`
- `(your |my )?(classmates|peers|cohort) (are|have|find|tend|struggle|...)`
- `(another|other) (student|learner) (named|called)`
- `one of your (classmates|peers|fellow learners|fellow students)`
- `(many|most|several|some|all|other) (students|learners|classmates|peers) (in this|are|have|...)`
- `\d{1,3}% of (students|learners|your classmates|the class|the cohort)`
- `the (class|cohort|other learners) (typically|usually|often|tends? to|...)`

## What passes deliberately

The validator is conservative on two specific cases that look similar but are not leaks:

- **Same-session backreferences** — "Earlier in this chat you asked about stoichiometry" is fine. The model has the current conversation in context.
- **Definition-style mentions of "classmates"** — "Classmates is a synonym for fellow students" is fine. The leak is in the verb phrase, not the noun.

Both are pinned as `expect: pass` in the corpus.

## Corpus expansion

`tests/security/memory_leak/` adds 6 fixtures: 4 fail-expected (one named-classmate, one aggregate-claim, one false-memory, one named-classmate-singular) plus 2 pass-expected (clean topic answer, in-session backreference). All run in `php admin/cli/run_validators.php` with the rest of the static corpus.

## PHPUnit coverage of the validator class itself

`tests/memory_leak_validator_test.php` adds 10 unit tests asserting:

- `name()` returns the stable machine ID `memory_leak`.
- Clean tutoring text passes.
- Each threat-surface family fails with the right `kind` in details.
- Same-session backreferences pass.
- Word "classmates" alone (no verb context) passes.
- Hit details include the matched phrase for incident review.
- Multi-pattern outputs collect both `false_memory` and `other_learners` kinds.

- Jailbreak: 32/32 unchanged — already covered the surface live since v5.3.24.

## Backwards compatibility

Drop-in upgrade from v5.3.34. Six new files (1 validator class, 1 PHPUnit file, 6 corpus fixtures) plus a two-line registration in `admin/cli/run_validators.php`. **No behavior change to the SSE active path** — the validator is registered and tested but not yet wired into `runtime_guard` for live blocking. That wiring is a separate change with a deliberate roll-out plan and is out of scope for v5.3.35.

Plugin version `2026050835`, release `5.3.35`.
