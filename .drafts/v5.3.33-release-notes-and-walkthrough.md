# SOLA v5.3.33

Cron task PHPUnit coverage **batch 3 of 3** completes 17 of 17 SOLA scheduled tasks tested for the first time in plugin history. Plus a **static-analysis test** that catches the `?:` numeric-default pattern automatically — and **8 more production bugs** of that pattern that the new test surfaced beyond what the manual sweep had caught.

## The shape of the bug

```php
// BAD — what the codebase had in 13 places.
$days = (int) (get_config('local_ai_course_assistant', 'audit_retention_days') ?: 365);
```

PHP's `?:` operator treats the string `"0"` as falsy. So an admin who sets the config to literal `0` to disable the feature gets the 365-day default silently re-applied. Every documented "set to 0 to disable / unlimited / corner-snap / always-trigger" contract was broken.

```php
// GOOD — uniform fix across all 13 sites.
$raw = get_config('local_ai_course_assistant', 'audit_retention_days');
$days = ($raw === false || $raw === '') ? 365 : (int) $raw;
```

## The static-analysis test

`tests/cron_tasks_test.php::test_no_get_config_falsy_default_pattern_in_classes` walks every `.php` under `classes/` and regex-matches `get_config(...) ?: <numeric|float>`. Any hit fails the suite with file:line and the offending default.

It's a hard CI gate now. Any future copy-paste of the bad pattern fails the suite immediately, not 6 months later when an admin hits the silent re-default.

I built the test in this release because manually grepping caught 5 of the bugs but missed 8. Same pattern, different files. The static check is what should have existed from the beginning.
