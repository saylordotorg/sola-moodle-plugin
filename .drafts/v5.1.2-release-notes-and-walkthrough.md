# SOLA v5.1.2

Hotfix release. Two TTS fixes that together eliminate the "read aloud takes 30 seconds to start" symptom.

## What's fixed

**Chunked TTS path was silently broken.** A function-name typo in `amd/src/chat.js` referenced `getOrCreateAudioContext` (undefined) where `getOrCreateAudioCtx` is defined. Every long-text read-aloud click threw a ReferenceError. The chunked path either died on the spot, or fell through to browser TTS, which on Chrome desktop can take 30+ seconds to initialize voices on first use. The non-chunked path (short text under 220 characters) was unaffected because it accessed the audio context directly through the module-level `sharedAudioCtx` variable.

Renamed the two references to match the helper definition. Chunked path now works as designed — time-to-first-audio drops from 10-20 seconds to 1-3 seconds on long assistant responses.

## What's new

**Visible loading state on the speak button.** A subtle pulse animation runs between dim and full opacity from the moment the button is clicked until first audio actually plays. Cleared on:
- play start (audio is now audible)
- explicit stop (learner cancelled)
- decode failure or any other error (so the indicator never gets stuck)

Learners running with `prefers-reduced-motion` get a static dim state rather than a pulse.

This means even on a healthy provider where TTS responds in 2-3 seconds, the learner sees immediate feedback that their click registered, instead of wondering whether the system is doing anything at all.

## Backwards compatibility

Drop-in upgrade from v5.1.1. No setting changes, no schema changes, no behavioural changes outside the read-aloud path. Two affected files: `amd/src/chat.js` (and its rebuilt `amd/build/chat.min.js`), and `styles.css`.

Plugin version `2026050502`, release `5.1.2`.
