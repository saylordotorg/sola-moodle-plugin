# SOLA v5.4.4

Provider benchmark — automated admin test that sends a fixed set of typical SOLA prompts to every configured AI provider and reports token usage, cost, latency, and a recommendation per capability (chat / analytics / voice / RAG). Results are exportable as JSON, CSV, Markdown, or plain text.

Single-provider sites are first-class: the run still produces a full result table and the recommendation is "this is the only provider you have." Multi-provider sites get the cheapest provider that completed every prompt; cost ties break on latency.

## Why

Until this release, an admin who wanted to compare two LLMs had to:
1. Configure the second one as a `comparison_providers` row.
2. Drive traffic through both manually.
3. Pull tokens and cost out of `mdl_local_ai_course_assistant_msgs` by hand.

That's a multi-hour exercise nobody actually does. v5.4.4 collapses it into a single click (admin UI) or one CLI invocation, runs the same prompts against every configured provider, and returns the comparison + a recommendation in seconds.

## The 11 prompts

The benchmark is intentionally small and zero-context — this is a cost / latency / contract benchmark, not a tutoring-quality benchmark. Five typical learner prompts (chat) plus three analyst-shaped prompts (analytics) plus three sample-passage embeddings (RAG):

**Chat (5)**

1. "What is photosynthesis? Answer in two sentences."
2. "Explain Newton's second law in one sentence."
3. "Give a concrete example of supply and demand."
4. "I finished the unit on cell biology. What's a good next concept to study?"
5. "Summarize what a quadratic equation is, in 2 sentences."

**Analytics (3)**

1. "How would you measure active-learner engagement on a Moodle site?"
2. "What are common sticking points first-year STEM students hit?"
3. "Briefly: how do you balance LLM cost against response quality at scale?"

**RAG (3)** — embedded passages, not prompts: a photosynthesis paragraph, a supply-and-demand definition, a binary-search description.

**Voice** — config snapshot only; Realtime is WebSocket and auth is per-session ephemeral, so a live test isn't meaningful at this layer.

## Output

Per-capability table with one row per (provider, model, prompt):

| Provider | Model | Prompt | OK | Prompt tok | Compl tok | Cost (USD) | Latency (ms) | Response excerpt |

Plus a recommendation line per capability: *"Lowest total cost ($X) across N prompts; latency Yms"* OR *"Only provider that completed every prompt"* OR *"Cost data unavailable; lowest total latency"*.

## Exports

The full payload is exportable as:

- **JSON** — pretty-printed, suitable for Redash, audit retention, or piping into `jq`.
- **CSV** — header row + per-prompt rows + a `# Recommendations` summary block at the end. Works in Excel / Google Sheets directly.
- **Markdown** — section per capability with tables and a recommendation line. Renders cleanly in GitHub issues, Confluence, or Slack snippets.
- **Plain text** — markdown with the table pipes stripped. For text/plain emails or terminals that don't render markdown.

Both the CLI and the admin UI offer all four. The admin UI shows last-run results inline (Markdown rendered via Moodle's `format_text`) and exposes per-format download buttons.

## Live first-run

Real OpenAI calls landed on the local dev install on the first run — 5 chat + 3 analytics + 3 RAG = 11 API calls, all completed, recommendation table rendered. Cost showed `—` because the OpenAI provider's non-streaming `chat_completion` path doesn't expose `get_last_token_usage()` uniformly; the tool falls back to latency-based recommendation in that case. Cost numbers will populate when the providers report tokens via the streaming interface (most do).

## Usage

CLI:

```bash
# Markdown to terminal (default)
php admin/cli/provider_benchmark.php

# JSON for piping
php admin/cli/provider_benchmark.php --format=json

# CSV to file
php admin/cli/provider_benchmark.php --format=csv > /tmp/sola-benchmark.csv

# Course-scoped (uses per-course provider override if set)
php admin/cli/provider_benchmark.php --courseid=2
```

Admin UI: navigate to `/local/ai_course_assistant/provider_benchmark.php` (capability-gated to `moodle/site:config`). Click "Run benchmark now" — full run takes 30–90 seconds depending on how many providers are configured. Once complete, last-run results render inline plus per-format download buttons.

## Cost

Each run makes one API call per (provider × prompt) pair. With one provider configured, a full run is roughly 10–15 cents. With two providers, double. Practical impact: admins can run this comfortably once per week or after every provider config change.

## PHPUnit coverage

`tests/provider_benchmark_test.php` — 14 tests, 66 assertions:

- Provider enumeration: primary first; comparison rows surfaced; deduped on (id, model).
- Happy-path runs against the v5.3.26 stub provider for both chat and analytics.
- Latency recorded per prompt regardless of outcome.
- Note returned when no chat provider is configured.
- Recommendation logic: single provider, lowest cost wins, providers that didn't complete every prompt are excluded, null when zero complete.
- Exports: JSON round-trips through decode; CSV header + data rows + recommendation block; Markdown contains all four section headers; plain text strips markers.
