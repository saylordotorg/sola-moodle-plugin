# SOLA v5.3.37

**Mastery threshold tuning fix.** Lowers `DEFAULT_THRESHOLD` from 0.85 to 0.75. The previous default was **mathematically unreachable** — every learner using mastery tracking in production with default settings was stuck at `learning` forever. Surfaced during the v5.3.34 mastery-test work; user-facing functional bug, fixed here as Option A from the four design alternatives.

## The bug

With the constants `PRIOR_ALPHA=2 / PRIOR_BETA=2 / DEFAULT_DECAY=0.9 / DEFAULT_WINDOW=8`, the Beta-binomial estimator caps the achievable score at approximately **0.79**, even for 8 consecutive perfect attempts at full weight. The pre-v5.3.37 default threshold of `0.85` could never be crossed.

The achievable score curve for N consecutive perfect attempts:

| N attempts | Decay-weighted sum W | Score = (2 + W) / (4 + W) |
|---|---|---|
| 1 | 1.000 | 0.60 |
| 2 | 1.900 | 0.66 |
| 3 | 2.710 | 0.70 |
| 4 | 3.439 | 0.73 |
| **5** | **4.095** | **0.75** ← new threshold |
| 6 | 4.686 | 0.77 |
| 7 | 5.217 | 0.78 |
| 8 (window cap) | 5.695 | **0.79** ← maximum achievable |
| ∞ (no cap) | 10.000 | 0.86 |

Mastery tracking is in production at every Saylor partner site running SOLA with the feature enabled. Until this release, none of those learners could ever flip from `learning` to `mastered` regardless of how perfectly they practiced. The product silently didn't deliver one of its advertised states.

## The fix

`DEFAULT_THRESHOLD` 0.85 → 0.75. With this single constant change:

- **5 consecutive perfect attempts** now promotes to `mastered` (score ≈ 0.753).
- **4 perfect attempts** stays at `learning` (score ≈ 0.731 — boundary).
- **Any wrong answer within the 8-attempt window** drops below 0.75 and blocks mastery. The wrong attempt has to age out of the window before promotion can fire. That is intentional: `mastered` is "perfect recent practice," not "mostly correct in the long run."
- **8 perfect attempts** sits at the window ceiling (≈ 0.794) — the math doesn't move further with more attempts because of the geometric decay.

## Migration is conservative

`db/upgrade.php` adds a savepoint at version `2026050837` that updates the stored `mastery_threshold` config from 0.85 → 0.75 **only when the existing value is the legacy default**. Admin customisations to anything other than 0.85 (e.g., 0.80, 0.70) are preserved. Callers who explicitly set 0.85 ARE migrated; that is intentional since 0.85 is the broken value.

No data migration. Mastery is computed on read in `compute_mastery()`, so existing learners' status updates the next time their dashboard or quiz triggers a recompute. No backfill needed.

## What's NOT changing

- `PRIOR_ALPHA = 2.0`, `PRIOR_BETA = 2.0` — the Beta-binomial smoothing prior is preserved. It was deliberately tuned to handle small-sample noise: 1 correct attempt → 0.60 (not jumping to 1.0), 1 wrong → 0.40 (not jumping to 0.0). Weakening the prior would change the noise floor; that's a separate decision not in scope here.
- `DEFAULT_DECAY = 0.9` — recency bias. Older attempts count less. Tuning this changes how forgiving mastery is to one-off bad days; out of scope.
- `DEFAULT_WINDOW = 8` — how many recent attempts feed the estimator. Geometric decay means widening this past 12 has minimal marginal effect; out of scope.
- `MIN_ATTEMPTS_FOR_MASTERY = 3` — anti-luck floor. A single perfect attempt can't promote.

## Why Option A and not the alternatives

| Option | Change | Perfect attempts to mastered | Why not |
|--------|--------|-----|---------|
| **A** (this release) | THRESHOLD 0.85 → 0.75 | 5 | Cleanest single-constant change. |
| B | PRIOR α/β: 2/2 → 1/1 | 6 | Disturbs small-sample noise smoothing. Tighter mastery, but 1 correct attempt jumps from 0.60 to 0.67 — louder early signal. |
| C | WINDOW: 8 → 20 | 20+ | Doesn't help: geometric decay caps marginal contribution. Would only matter for learners with extreme attempt volumes. |
| D | Combine A + DECAY 0.9 → 0.95 | 5 with longer retention | Mastery becomes "stickier" once earned. Useful but not the primary problem here. Defer. |

## Test pins

`tests/mastery_test.php` adds 4 tests that pin the new curve. These will fail-loud if any future tuning regresses to "mastery is unreachable":

- `test_default_threshold_five_perfect_attempts_promotes_to_mastered`
- `test_default_threshold_four_perfect_attempts_stays_at_learning`
- `test_default_threshold_one_mistake_in_window_blocks_mastery`
- `test_default_threshold_eight_perfect_attempts_reaches_window_ceiling`

Each test calls `unset_config('mastery_threshold', ...)` first so the assertion exercises the constant default, not whatever was stored at install time. The savepoint handles the install-time question for production sites.

## Backwards compatibility

Drop-in upgrade from v5.3.36. Three production code files modified (`objective_manager.php` constant, `settings.php` admin default, `db/upgrade.php` savepoint), one new pinning test, no schema changes.

**Existing learners who had been silently stuck at `learning` despite consistent perfect practice will start showing as `mastered` after this upgrade.** That is the intended behaviour. Admin dashboards that count "% mastered per objective" will see the metric jump on the first compute_mastery call after deploy — that's a real change in observable system state, not a bug.

Plugin version `2026050837`, release `5.3.37`.
