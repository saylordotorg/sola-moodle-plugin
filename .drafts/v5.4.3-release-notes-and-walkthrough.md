# SOLA v5.4.3

Universal email unsubscribe mechanism. Every SOLA email pathway — learner-facing AND admin/staff — now appends an unsubscribe footer with a one-click HMAC-signed link, and every send checks the recipient's opt-out state before delivering. Trigger: unsubscribe was missing on the Learning Radar Scheduled email; on review six other SOLA email types had the same gap. The fix is uniform.

## How the universal mechanism works

1. **Token mint.** `email_optout::url($email, $type)` returns a `/local/ai_course_assistant/email_unsubscribe.php?token=<hmac-signed>` URL. Token format: `b64url(email|type|expiry).b64url(hmac_sha256(payload, $CFG->siteidentifier))`. 60-day expiry. Same shape as the existing `digest_unsubscribe_token` (v4.1) for consistency.

2. **Footer composition.** `email_footer::append_text($body, $email, $type, $reason)` and `email_footer::append_html(...)` append a small footer to outgoing bodies. The reason line is optional and per-call-site so each pathway can explain why the recipient is on the list.

3. **Pre-send opt-out check.** Each email pathway calls `email_optout::is_opted_out($email, $type)` before sending. Opted-out recipients are silently skipped (logged where the pathway has mtrace). Idempotent insert via the unique key on `(email, optout_type)` so duplicate link clicks are a no-op.

4. **Endpoint.** `email_unsubscribe.php` validates the HMAC token, writes the opt-out row, and either renders a confirmation page (GET) or returns 200 silently (RFC 8058 List-Unsubscribe-Post for native mail-client buttons).

## Schema

`db/install.xml` + `db/upgrade.php` (savepoint `2026050843`):

```
local_ai_course_assistant_email_optout
  id           int  PK
  email        varchar(190)  NOT NULL
  optout_type  varchar(60)   NOT NULL
  userid       int  NULL  (Moodle user id when known)
  courseid     int  NULL  (course context when known)
  timecreated  int  NOT NULL
  UNIQUE  (email, optout_type)
  INDEX   (userid)
  INDEX   (optout_type)
```

The `(email, optout_type)` unique key handles both Moodle-user destinations and arbitrary admin/staff email addresses uniformly. `userid` and `courseid` are nullable — they're recorded for diagnostic queries but not part of the lookup key.

## PHPUnit coverage

`tests/email_optout_test.php` — 17 tests, 34 assertions:

- **Storage** (7): unknown-pair returns false; record-then-check round-trip; idempotent record on duplicate; case+whitespace normalisation; per-type isolation; clear removes opt-out; empty email/type is a no-op.
- **Token** (5): mint→verify round-trips email + type; tampered signature rejected; expired token rejected; malformed token rejected; URL contains a signed token.
- **Footer composition** (5): text footer includes URL + token + call-to-action; reason line surfaces; HTML footer renders an anchor; `append_text` concatenates; full round-trip — the token in a generated footer verifies back to the original `(email, type)`.

## Backwards compatibility

Drop-in upgrade from v5.4.2. New table created via savepoint. No breaking changes to existing pathways. Recipients with no opt-out row see no behaviour change other than the new footer appearing on emails that previously had no footer (the 8 pathways above).

Pre-existing affordances (`digest_unsubscribe_token` and `reminder_manager`'s per-row token) are unchanged — sites already using those keep working.

Plugin version `2026050843`, release `5.4.3`.
