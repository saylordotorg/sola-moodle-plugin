# SOLA v5.4.5

Production security hardening for the saylor.org 30-course pilot. Triggered by Tom's pre-rollout audit ask: dependencies pinned + SCA scanned, no secrets in repo, outbound LLM keys scoped, logs PII-clean, documented emergency disable.

Outcome: four of the five asks were already in good shape (logging, secrets, scoping mechanism); the missing pieces were operational artifacts (Dependabot config, scoping documentation, emergency-disable CLI, incident response runbook). All shipped in this release.

## The emergency-disable mechanism

`php admin/cli/emergency_disable.php` (no flags) takes SOLA fully offline:

- `enabled=0` — chat widget stops rendering, scheduled tasks short-circuit, SSE rejects.
- `voice_active_realtime=''` — Realtime token-mint endpoint refuses (with the prior value stashed into `voice_active_realtime_backup` for restore).
- `rag_enabled=0` and `rag_auto_reindex_drifted=0` — retrieval and indexing both stop.
- `outreach_master_enabled=0` — milestone, streak, and digest emails stop.
- An `emergency_disable` audit row is written with the invoker, flags, and reason.
- `purge_all_caches()` is called so the change is immediate on the very next request.

Subsystem flags scope the kill: `--voice`, `--rag`, `--outreach`, `--chat` (chat-only sets `spend_cap_site=0` so the existing budget-paused path returns a friendly message to learners).

Reverse with `--restore` — voice and chat backups round-trip exactly to their prior values.

The full runbook (decision tree, time-bound steps, audit-log queries, post-mortem template) is in `.wiki/Security-Incident-Response.md`.

## What was already good

The pre-audit findings worth flagging because they're load-bearing for the pilot:

- **Logging is PII-clean.** Zero `debugging()` / `error_log()` / `mtrace()` calls emit user content. The audit log records `message_length` (an integer) on `message_sent`, never the text.
- **Secrets in the repo: none.** `wrangler.toml` for the Cloudflare Worker explicitly documents `wrangler secret put XAI_API_KEY` / `MOODLE_JWT_SECRET` — the master xAI key never enters the repo.
- **GDPR Article 15 + 17 already covered.** The privacy provider declares 21 database_table items (per the v5.4.2 health check). The v5.3.17 + v5.3.18 fixes from earlier this month closed both export and delete coverage gaps.
- **Spend caps + per-validator runtime guard.** `spend_cap_site` enforces budget at request time; the v4.8.0 runtime_guard runs `pii_echo` + `credential_leak` + `hallucination` + `second_person` on every response and the v5.4.0 `memory_leak_validator` is wired in behind a feature flag for staged rollout.

## Production hardening checklist

`SECURITY.md` now includes a 9-item checklist for every production install:

- [ ] Provider API keys are scoped per `.wiki/Provider-Key-Scoping.md`.
- [ ] Provider keys rotated within 90 days and on every staff turnover.
- [ ] `spend_cap_site` set to monthly LLM budget × 0.9.
- [ ] `audit_retention_days` set to institutional policy (default 365).
- [ ] `conversation_retention_days` set to FERPA / GDPR Article 5 storage limit (default 730).
- [ ] `validators_runtime_mode` is `annotate` or `block`, not `off`.
- [ ] `php admin/cli/health_check.php --strict` exits 0 immediately after deploy.
- [ ] Dependabot enabled.
- [ ] Emergency-disable runbook reviewed by the on-call admin.

## PHPUnit coverage

`tests/emergency_control_test.php` — 10 tests, 21 assertions:

- Full kill disables master switch + RAG + outreach.
- Voice kill stashes `voice_active_realtime` into `_backup` then clears.
- Voice restore puts the stashed value back exactly.
- Chat kill uses `spend_cap_site=0` (with backup); chat restore round-trips.
- Disable + restore each write an audit row with the invoker and reason.
- Unknown flags are tolerated (defensive against partner-tooling drift).
- Subsystem flags do not touch the master `enabled` switch.
