# SOLA v5.4.0

End-of-day release. Closes both remaining items from the test-coverage sweep and bumps the minor version to mark the milestone — **all six LLM-calling external services now have unit-test coverage, all 17 scheduled tasks have unit-test coverage, the jailbreak suite has a static-corpus pair via `memory_leak_validator`, and the validator is wired into the live response path under a feature flag for staged roll-out.**

## What's in this release

### 1. `memory_leak_validator` wired into `runtime_guard`

The validator class was added in v5.3.35 with a static-corpus regression suite, but it had no live-path exposure. v5.4.0 adds it to `runtime_guard`'s active SSE response chain behind a separate feature flag for safe staged rollout:

- New admin setting: `validators_runtime_memory_leak_enabled`. **Default: off.**
- When the flag is on AND `validators_runtime_mode` is `annotate` or `block`, the validator joins the chain alongside `pii_echo`, `credential_leak`, `hallucination`, and `second_person`.
- When the flag is off (default), `runtime_guard` runs the original four validators only — no behaviour change for existing installs.
- Audit logging via `audit_logger::log('runtime_validator_fail', ...)` records `memory_leak` in the `validators` array of the details JSON, so the ops team can monitor false-positive rates after enabling.

Recommended staged rollout:
1. Enable the flag with `validators_runtime_mode=annotate` on one dev site.
2. Watch `local_ai_course_assistant_audit` for `runtime_validator_fail` rows tagged `memory_leak` — false positives appear as legitimate-looking responses with the warning line.
3. Tune the regex in `memory_leak_validator.php` if the false-positive rate is non-zero.
4. Promote `validators_runtime_mode` to `block` once the false-positive rate stabilises near zero.

### 2. HTTP-client wrapper for `get_realtime_token`

The OpenAI Realtime token-mint endpoint was the last LLM-calling path without happy-path coverage — its curl call to `api.openai.com/v1/realtime/client_secrets` was uncoverable without a real API key. v5.4.0 adds the cheapest possible "DI lite" hook:

- `get_realtime_token::$test_http_response` — public static array. When set to `['body' => ..., 'http_code' => ...]`, short-circuits the curl call and returns the canned response. Production paths leave it null at boot.
- The curl logic moved into `call_openai_realtime()` private helper. Production behaviour identical to v5.3.x.
- Tests reset the static in `tearDown()` so a leak from one test cannot cross into another or, worse, into a non-test path.

Six new tests in `tests/realtime_token_test.php` exercise the mocked path:

- Happy path: 200 + nested `client_secret.value` returns the minted token.
- Legacy shape: 200 + top-level `value` field also works (older OpenAI Realtime API responses).
- 401: `Incorrect API key provided` from upstream surfaces in the thrown exception message.
- 500: generic fallback message when no `error.message` field is present.
- 200 with empty token: rejected with `Unexpected response`.
- `clean_returnvalue` round-trip on the OpenAI path.

### 3. Test counts

`tests/runtime_guard_test.php` — new file, 8 tests:

- Off mode returns response unchanged.
- Annotate mode appends review line on credential_leak hit.
- Block mode replaces response with safe fallback.
- Memory-leak flag off does not apply validator.
- Memory-leak flag on annotates false memory narration.
- Memory-leak flag on + block mode replaces response.
- Memory-leak flag on + clean response unchanged.
- Memory-leak flag on writes audit row with validator name.

PHPUnit total: **253 tests** (was 239, +14 in this release across the two new files).

The 19 production bugs caught + fixed during the sweep:

- **18 `?: <numeric|float|::CONSTANT>` fall-through bugs** across 15 files, all the same shape — PHP's `(int)"0"` is falsy so `?: <default>` silently re-applied the default whenever an admin set a config to literal `0`. Affected configs: `audit_retention_days`, `conversation_retention_days`, `inactivity_threshold_days`, `anomaly_digest_threshold_pct`, `temperature` (×2 providers), `prompt_budget_chars` (×2 sites), `offtopic_max_per_session`, `profile_update_interval`, `survey_trigger_messages`, `position_offset_x`, `position_offset_y`, `rag_chunksize` (×2 sites), `max_content_per_resource`, `max_total_content`, `maxhistory`, `embed_dimensions`, `rag_topk`, `mastery_threshold`, `mastery_window`, `mastery_decay_half_life_days`, `attachment_max_size_mb` (×2 sites). Now caught at CI-gate time by `cron_tasks_test::test_no_get_config_falsy_default_pattern_in_classes`.
- **1 unreachable-mastery threshold tuning bug** (v5.3.37). `DEFAULT_THRESHOLD=0.85` was mathematically unreachable with the rest of the constants — every learner using mastery tracking was silently stuck at `learning`. Lowered to 0.75; existing installs migrated via `db/upgrade.php` savepoint that respects admin customisations.

## Open follow-ups

The day's sweep is complete. Three items intentionally left for separate decisions:

1. **Promote `validators_runtime_memory_leak_enabled` to default-on** after a staging-period observation of false-positive rates on dev.sylr.org. ~1 release. Not blocked, just deferred until there's enough data to be confident the regex is sound.

2. **Mastery threshold tuning observation** — the v5.3.37 fix took the cleanest single-constant approach (Option A from four alternatives). Once a few months of post-deploy data accumulates, it would be worth eyeballing the actual `% of active learners reaching mastered per objective` against intent. If the rate is too low or too high, the next move is one of: lower `DEFAULT_THRESHOLD` further, weaken the Bayesian prior, or expand the window.

3. **Behat coverage of the quiz flow + analytics dashboard + course settings + RAG admin**. Each is one or two scenarios with the v5.3.26 stub provider. Quiz flow is the highest-value (catches coach-mode regressions, scoring math, next-question routing). Each is independent half-day work.

## Backwards compatibility

Drop-in upgrade from v5.3.37. Two production code files modified:

- `classes/runtime_guard.php` — adds memory_leak validator to chain only when the new feature flag is set.
- `classes/external/get_realtime_token.php` — extracts curl call to a private helper plus a test-only override hook.

No schema changes, no string changes. **The new feature flag defaults off, so existing installs see zero behaviour change until an admin opts in.**

Plugin version `2026050840`, release `5.4.0`.
