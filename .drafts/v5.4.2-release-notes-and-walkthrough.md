# SOLA v5.4.2

Post-deploy health check CLI. Designed for partner ops teams to wire into their deploy procedure as the last step — exit 0 means the upgrade is healthy, exit 1 means at least one check failed and the deploy should be investigated or rolled back before traffic resumes.

## Why this exists

A partner-side upgrade has at least these silent failure modes that none of our existing tests catch:

- Plugin files rsynced but `db/upgrade.php` interrupted mid-savepoint.
- Schedule rows for new SOLA tasks not inserted (cron silently never fires them).
- Language string keys referenced in code but not present in the partner's locale (learner sees `[[chat:placeholder]]`).
- Privacy provider not registering all SOLA tables (GDPR export silently omits learner data).
- A SOLA cron task in instant-fail loop, growing its `faildelay` past safe bounds (the same pattern that surfaced on degrees.saylor.org with `enrol_programs`).
- A hand-edit on the deployed server reintroducing the `?:` numeric-default pattern.

`admin/cli/health_check.php` runs in ~1 second, exits with a clean 0 or a meaningful 1, and is safe for every partner to wire into their deploy gate.

## The 8 checks

| # | Name | What it verifies |
|---|------|-------------------|
| 1 | `plugin_version_vs_db_savepoints` | DB version matches/exceeds `version.php` — catches half-applied upgrades |
| 2 | `scheduled_tasks_registered` | Every `classes/task/*.php` (non-adhoc) has a row in `mdl_task_scheduled` |
| 3 | `lang_strings_load` | 10 representative lang keys return non-placeholder strings |
| 4 | `privacy_provider_metadata` | Privacy provider declares at least one `database_table` (GDPR coverage) |
| 5 | `no_cron_tasks_stuck` | No SOLA cron task has `faildelay > 3600s` (the `enrol_programs` failure mode) |
| 6 | `audit_log_writable` | Insert + select + delete round-trip on the audit table works |
| 7 | `no_falsy_default_get_config_pattern` | Static-analysis re-check of the v5.3.33 `?:` pattern against the deployed `classes/` |
| 8 | `provider_resolves_when_enabled` | If `enabled=1` and `provider` set, the factory instantiates without throwing |

## Real catch on first run

The CLI immediately caught a real issue on my local dev Moodle: DB version was at the v5.3.0 savepoint (`2026050700`) but `version.php` was at v5.4.1 (`2026050841`). My rsync-deploy-and-purge cycle today never ran `admin/cli/upgrade.php` to apply the v5.3.37 mastery-threshold migration savepoint. After running upgrade, all 8 checks went green.

This is exactly the silent-regression failure mode the check is designed for — without it, the local install would have continued shipping with a stale schema for as long as I kept using the rsync-only pattern.

## Usage

```bash
php admin/cli/health_check.php          # run, exit 0 on full pass
php admin/cli/health_check.php --strict # warnings count as failures
php admin/cli/health_check.php -v       # show one-line message per check, even passes
```

Recommended setting for automated deploy gates: `--strict`.

## PHPUnit coverage of the checks themselves

`tests/health_check_test.php` — 15 tests, 61 assertions:

- `run_all` returns the documented summary shape.
- A clean PHPUnit install passes every check.
- Per-check negative tests: missing version row, DB-behind-version.php, unregistered task, stuck cron with `faildelay=7200s`, plugin-enabled-with-no-provider (warns), unknown-provider-id (fails).

The negative tests pin the failure detection — a future refactor that breaks one of the checks in a way that returns PASS on a broken install would surface here at CI time.
